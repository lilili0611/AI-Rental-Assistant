## 0. v2.0 实现变更记录 (相对 v1.0 的已落地差异)

> 本节汇总 Phase 1+2 实际开发中确认的决策与对 v1.0 的偏离。后续迭代以本节为准。
> **铁律：每次迭代必须先更新本 PRD 和 Spec，再改代码。**

1. **品牌**：相机租赁 → **猫猫头相机租赁**；前端为「活泼创意」风格 + 猫猫卡通吉祥物。
2. **定价模型彻底变更**（覆盖原 3.1）：业务方提供真实价格表，改为**档位计价**（两天价 / 三天价 / 三天以上续租单价），**完全取代**原"日租 × 天数阶梯折扣 × 季节折扣"。详见新版 3.1。
3. **真实设备目录已上线**：11 设备 / 13 配置 / 15 台实物，含各自押金。详见 3.8。
4. **权限**：Phase 1+2 采用简化三级 `customer / staff / admin`（role 字段可扩展到 5 角色）。
5. **认证（MVP）**：手机号直接登录（暂不接短信验证码），返回用户标识。
6. **支付**：确认为人工线下收款 + 商家后台「确认收款」推进状态，**无在线支付**（与 v1.0 一致，已澄清）。
7. **技术栈落地**：SQLite 起步（架构可迁移 PostgreSQL）；LLM 用 DeepSeek（OpenAI 兼容，无 Key 时降级关键词规则）；用 APScheduler 替代 Celery+Redis。详见 Spec。
8. **飞书双向同步已实现**：AI→飞书写入 + 飞书→AI 轮询回流（含冲突策略）。详见 5.2 与 Spec 第 7 章。
9. **租客自助下单前端已上线**：浏览设备→选配置/租期→查库存算价→下单→我的订单/取消 + AI 聊天侧栏。
10. **已知缺口（功能未做，非缺陷）**：租客订单详情/收款引导页、商家端「确认收款/发货」管理界面（后端接口已具备）。→ v2.1 已纳入：见 §0.1，商家管理页本次实现。

## 0.1 v2.1 变更记录 — 订单审核与物流流程 (本次迭代)

> 本节定义本次迭代「做什么、为什么」，技术实现见 Spec v2.1 同名章节。
> **铁律：本节先于代码定稿。**

**背景**：客人在前端下单后，缺少一个面向商家的「核对是否已付款 → 放行档期 → 发货填物流 → 验收完结」的闭环；商家此前只能靠飞书表或裸调接口操作。本次补齐客户可见的订单流转标签、物流单号展示，以及一个简单的商家管理页。

**1. 设计原则：内部状态机不变，新增「客户可见标签」映射层。**
现有内部状态（`pending_payment/paid/confirmed/shipped/...`）保持不变，以兼容已有飞书同步与测试；客户与商家界面只展示映射后的中文标签。映射关系（详见 Spec §3.1）：

| 内部状态 | 客户/商家可见标签 |
|---------|------------------|
| pending_payment / paid | **商家审核中** |
| confirmed | **已确认档期（待发货）** |
| shipped | **已发货**（展示快递公司 + 物流单号） |
| completed | **订单已完结** |
| cancelled | 已取消 |

> 旧数据中可能出现的 `active/returned` 仍保留定义，但 v2.1 默认流程不经过它们（见原则 3）。

**2. 商家审核 = 一步完成「确认收款 + 放行档期」。**
客人下单后订单为「商家审核中」。商家在管理页核对是否已收到款：
- **审核通过**：一步把订单推进到「已确认档期（待发货）」，同时记录已收金额与收款备注。
- **审核驳回**：订单留在「商家审核中」，附驳回原因（如未收到款），客户侧可见提示。

**3. 发货后直接由商家验收完结，不经客户签收/归还。**（业务方确认）
流程为：已确认档期 →（商家上传物流）→ 已发货 →（商家验收）→ 订单已完结。**新增状态转换 `shipped → completed`**，跳过 `active/returned`。客户端不做签收/归还操作。

**4. 物流：快递公司 + 单号，手工录入，前端只读展示。**（业务方确认）
商家在管理页填写**快递公司**与**物流单号**，该操作同时把订单推进到「已发货」。前端客户订单展示这两项，不对接快递 API、无实时轨迹（实时轨迹留待后续，见 §7.1 外部依赖）。

**5. 商家管理页（本次实现）。**
新增一个简单的 staff 专用后台页：订单列表 + 每单的「审核通过 / 驳回」「上传物流并发货」「商家验收完结」操作按钮，以及当前状态与物流信息展示。鉴权沿用 `X-User-Id`（staff/admin 角色）。

**6. 飞书同步。**
新增「快递公司」「物流单号」两列纳入字段映射；订单状态文本归一表补充 v2.1 标签，使飞书侧人工改动可正确回流（详见 Spec §7）。

**7. 访问入口（临时记录，本地开发默认端口 8000）。**
启动命令 `uvicorn app.main:app --reload` 后：

| 入口 | 网址 | 说明 |
|------|------|------|
| 租客前端 | http://127.0.0.1:8000/ | 客户浏览设备 / 下单 / 我的订单 / AI 对话 |
| 商家后台 | http://127.0.0.1:8000/admin | 员工审核 / 上传物流 / 验收完结（手机号登录，需 staff/admin 角色） |
| API 文档 | http://127.0.0.1:8000/docs | Swagger 调试 |

> ⚠️ 以上为本地临时地址；部署到服务器后替换为正式域名。商家后台默认演示账号：手机号 `13900000002`（仓库小王，staff）。

## 0.2 v2.2 变更记录 — 商家后台密码鉴权 + 生产上线 (本次迭代)

> 对应 Spec v2.2 同名章节。**铁律：本节先于代码定稿。**

**背景**：v2.1 的商家后台 `/admin` 与租客前端共用「手机号直接登录」，没有口令——**任何人只要知道员工手机号就能进后台审核订单**。系统要对外（公网）开放给真实用户使用前，必须堵上这个洞，并完成生产部署改造。

**1. 商家后台改为「账号 + 密码」登录。**（业务方确认：后台加独立密码登录）
- 员工/管理员用**手机号 + 密码**登录后台，密码以**加盐哈希**存储（PBKDF2，不存明文）。
- 登录成功签发一个**带签名、带有效期的会话令牌（token）**；后台所有写操作接口校验该令牌且角色为 staff/admin。仅凭手机号或 user_id **不再能**操作后台。
- 租客前端维持手机号登录不变（租客只能看/操作自己的订单，风险较低；短信验证码仍列为后续增强）。
- 密码设置走线下脚本（管理员执行），系统不提供自助注册员工。

**2. 生产上线（国内云服务器，24 小时在线）。**（业务方确认：正式上线 + 国内）
- 目标形态：部署到国内「轻量应用服务器」，Nginx 反向代理 + HTTPS，进程守护常驻。
- **强约束**：国内服务器对外开放网站**必须完成 ICP 备案**（管局审核约 2–3 周），这是上线前置且不可加速。
- 密钥（DeepSeek Key、飞书凭证、`ENCRYPTION_KEY`）一律走**服务器环境变量**，不进代码库。
- 数据库起步沿用 SQLite（需挂持久化目录）；用户量上来后再迁 PostgreSQL（架构已兼容）。
- 后台定时任务（APScheduler）须保持**单实例**运行（不可多进程并发，否则预留扫描/飞书轮询会重复执行）。

**3. 角色与边界不变**：customer / staff / admin 三级；后台鉴权只收紧 staff/admin 入口。

> 部署细节（运行命令、Nginx/systemd 配置、环境变量清单、备案与域名步骤）见 Spec v2.2「第 11 章 生产部署」与仓库 `deploy/` 目录。

## 0.3 v2.3 变更记录 — 数据库迁移到 Supabase (本次迭代)

> 对应 Spec v2.3 同名章节。**铁律：本节先于代码定稿。**

**背景**：此前数据存在本地 SQLite 文件（`rental.db`），无法多端共享、云部署易丢失。改用 **Supabase（托管 PostgreSQL）** 作为正式数据库，数据集中、可云端访问、可备份。

**1. 数据库后端：本地 SQLite → Supabase Postgres。**
- 由于代码自始按「可迁移到 PostgreSQL」设计（SQLAlchemy ORM + `DATABASE_URL` 驱动、库存/计价等无 SQLite 专有逻辑），迁移**以连接配置 + 驱动为主，不重写业务代码**。
- 生产/云端 `DATABASE_URL` 指向 Supabase 连接串；**不再以本地 `rental.db` 为准**。
- 表结构由现有 ORM 模型在 Supabase 自动建立（与 Spec §2 数据模型一致，JSON 字段落 Postgres JSON/JSONB）。

**2. 本地→Supabase 数据迁移。**
- 提供迁移脚本，把本地 SQLite 现有数据（设备目录/配置/库存/用户/订单等）整体搬到 Supabase，作为初始测试数据。
- 亦可用 `scripts/seed_data.py` 直接对 Supabase 灌入干净的演示目录（二选一）。

**3. 测试与本地开发保持可用。**
- 单元测试仍用内存 SQLite（快、隔离），不连真实 Supabase；数据库可移植性正是靠 ORM 保证。
- 本地开发可继续用 SQLite 或连 Supabase，由 `DATABASE_URL` 切换。

**4. 凭证安全。**
- Supabase 连接串含密码，**只进环境变量 / `.env`，绝不入库**。建议用 Supabase 的连接池（Pooler）连接串以兼容 IPv4 与云平台。

> 技术细节（驱动、连接串格式、迁移脚本、连接池）见 Spec v2.3「§1.3 + 第 12 章」。

## 1. 产品背景与目标

### 1.1 业务现状与痛点

相机租赁业务当前依赖人工处理咨询、下单、库存登记、对账等环节。随着订单量增长,人工模式暴露出几个问题:客户咨询响应慢(尤其非工作时间)、库存信息分散在飞书表格中容易出错、订单状态靠人工跟进、损坏赔偿计算主观且易产生纠纷。

### 1.2 产品定位

一个**对接飞书数据、面向客户与内部员工的 AI 租赁管理助手**。客户侧提供 7×24 自助查询与下单;内部侧提供自动化的订单、库存、对账流程。核心是把标准化、重复性的业务流程交给 AI,把判断性、异常性的环节留给人工。

### 1.3 核心目标与成功指标

| 目标 | 衡量指标 | 目标值 (上线 3 个月后) |
|------|---------|----------------------|
| 提升响应效率 | 客户咨询平均响应时间 | < 10 秒 |
| 降低人工负担 | 标准流程 AI 自动处理占比 | > 80% |
| 减少库存差错 | 库存数据与实物不一致率 | < 2% |
| 保证数据一致 | AI 系统与飞书同步延迟 | < 1 分钟 |
| 控制赔偿纠纷 | 因赔偿标准引发的客诉率 | 环比下降 50% |

> ⚠️ **待确认**:上述目标值为基于行业经验的建议值,需结合当前实际业务量与人力成本校准。

## 2. 用户与角色

### 2.1 用户分类

系统服务两大类用户,权限与功能边界严格区分:

**C 端 — 租赁客户**
自助查询设备、库存、价格;创建和管理自己的订单;追踪物流。仅能访问与自己相关的数据,无法触碰内部库存配置、定价规则、其他客户信息。

**B 端 — 内部员工**
按职能划分为 5 种角色(详见下表),通过飞书或后台管理订单、库存、客诉、财务。

### 2.2 B 端角色权限矩阵

| 角色 | 核心职责 | 关键权限 |
|------|---------|---------|
| 销售员 | 接单、报价 | 查看全部数据、创建订单、生成报价 |
| 仓库管理 | 出入库 | 确认库存、发货操作、验收归还、损坏拍照记录 |
| 财务审核 | 资金 | 审核订单、确认收款、处理退款、生成对账单 |
| 客服 | 售后 | 处理投诉、生成工单、授权补偿 |
| 系统管理员 | 全局 | 全部操作、用户与权限管理、系统配置 |

> ⚠️ **待确认**:5 角色划分是基于通用租赁业务的设计。如果当前是小团队(例如老板 + 1-2 名员工身兼多职),Phase 1 可简化为"客户 / 员工 / 管理员"三级,5 角色推迟到团队扩大后。**这一点直接影响 Phase 1 的权限模块工作量,请优先拍板。**

### 2.3 认证方式

✅ **已确认**:用户认证优先使用**手机号**(短信验证码)。实名认证(身份证)作为租赁前置要求,涉及押金和设备安全。

> 🆕 **v2.2**:租客（C 端）维持手机号登录；**商家后台（B 端）改为「手机号 + 密码」登录**并签发会话令牌（见 §0.2 与 Spec §10/§11）。这是对外开放前的安全前置。

## 3. 业务规则 (核心,已确认)

本章是产品的业务内核,大部分已由业务方确认,是 AI 行为和系统计算的依据。

### 3.1 定价规则 ✅ (v2.0 已变更为档位计价)

> 🔴 **v2.0 变更**：业务方提供真实价格表，定价改为**档位计价**，**完全取代** v1.0 的"日租 × 天数折扣 × 季节折扣"。原天数阶梯折扣、季节折扣**已废弃**。

每个设备配置有三个价格字段：**两天租金 / 三天租金 / 三天以上续租单价(元/天)**。租赁天数 = 含起含止（9/1–9/3 记 3 天）。

**计价公式：**

| 租期天数 | 租金 |
|---------|------|
| 1–2 天 | 两天租金 |
| 3 天 | 三天租金 |
| > 3 天 | 三天租金 + (天数 − 3) × 续租单价 |

**总应付 = 租金 + 押金**（押金可退）。无折扣、无季节因素、无优惠券（优惠券留待 Phase 4）。

例：佳能 G12（两天100/三天120/续租25），租 5 天 = 120 + 2×25 = 170 元租金 + 押金。

> 各设备的三档价格与押金见 3.8 设备目录。

### 3.2 库存规则 ✅(但实现需修正)

✅ **已确认**:同一设备的不同配置(如「R5 + 16-35mm」与「R5 + 70-200mm」)**分别独立计库存**。

**库存占用是按租赁日期区间计算的** —— 这一点至关重要:某台设备在 9 月 1–3 日被租出,不影响它在 9 月 5 日的可用性。系统判断"某日期区间是否有货",必须检查该区间内**每一天**的占用情况。

> 🔴 **复盘提示**:Phase 1 已有代码采用的是全局计数(总数 − 已租 − 已预留),**无法正确处理按日期的库存**。这是 Spec 必须重新定义、开发必须修正的关键点。

**库存预留机制:** 客户下单后,系统预留库存 30 分钟。30 分钟内未完成支付确认,预留自动释放,库存回归可用。

### 3.3 支付规则 ✅(纠正前期设计)

✅ **已确认**:**支付由人工/第三方处理,AI 不自动扣款。**

这意味着系统中的"支付"环节是这样的:

1. AI 创建订单,状态置为"待支付"
2. 客户通过线下转账 / 第三方收款码 / 人工引导等方式付款(**不在本系统内完成**)
3. 财务或销售在飞书 / 后台**手动确认收款**,订单状态推进到"已支付"

> 🔴 **复盘提示**:前期设计文档与代码中出现的"生成支付二维码""支付宝/微信异步回调"等内容**与此确认矛盾,应全部移除**。系统只做支付状态的记录,不做支付通道的对接。

### 3.4 损坏赔偿规则 ✅(区间需校准)

赔偿按损坏类型和程度,以"扣除押金的百分比"计算。完整标准见下:

**机身/镜身划痕掉漆:**

| 程度 | 扣除 |
|------|------|
| ≤ 2mm | 押金 2% |
| 2–10mm | 押金 3% |
| 10–30mm | 押金 10% |
| > 30mm | 押金 15% |

**机身/镜身磕碰磨损:**

| 程度 | 扣除 |
|------|------|
| ≤ 1mm | 押金 3% |
| 1–3mm | 押金 5% |
| 3–5mm | 押金 10% |
| 5–10mm(外壳裂痕 ≤ 5mm) | 押金 15% |
| 10–20mm(外壳裂痕 ≤ 10mm) | 押金 20% |
| ≥ 20mm 凹痕 / ≥ 10mm 裂痕 | 押金 100% |

**镜片磨损:**

| 程度 | 扣除 |
|------|------|
| ≤ 1mm | 押金 5% |
| 1–5mm | 押金 10% |
| 5–10mm | 押金 20% |
| > 10mm 或出现镜片凹坑 | 押金 100% |

**其他:**

| 情况 | 扣除 |
|------|------|
| UV 镜变形或严重磨损 | 同 UV 镜价格 |
| 镜头卡口变形、对焦等性能故障 | 官方维修价 |
| 相机首次拆修 | 额外 10% 押金 |

> ⚠️ **待确认**:原始表格中部分区间边界存在歧义(例如"≤2mm"与"2–10mm"在 2mm 处、磕碰磨损里"≤3mm 全押金5%"与"3-5mm"的衔接)。Spec 中已将其整理为**左开右闭、无重叠**的精确区间,但需要业务方对照实物赔偿习惯做最终确认。

### 3.5 超期规则 ✅

租期结束未归还,按"超期天数 × 日租金 × 10%"计算滞纳金。处理节奏:第 1–3 天提醒并开始计费,第 7 天起进入催收流程。

### 3.6 取消规则 ✅

| 场景 | 费用 |
|------|------|
| 未支付,下单后 48 小时内 | 免费取消,释放库存 |
| 已支付,发货前 | 手续费 = 订单金额 10%,退还余额 |
| 已发货 | 不支持直接取消,需走拒收 / 人工 |

### 3.7 服务时效承诺 (SLA) ⚠️

| 环节 | 承诺 |
|------|------|
| 订单审核 | 24 小时内 |
| 发货 | 确认后 1–2 个工作日 |
| 验收 | 归还后 3–5 个工作日 |
| 对账 | 验收后 5 个工作日内 |
| 退款 | 申请后 3–5 个工作日 |

> ⚠️ **待确认**:SLA 是对客户的正式承诺,违约需补偿。请业务方确认这些时效是否符合实际履约能力,避免承诺过高。

## 4. 功能范围与 Phase 规划

本章是 PRD 的核心,按 Phase 拆解完整产品蓝图。每个 Phase 标注其功能、价值和确认状态。

### 4.1 总览

| Phase | 主题 | 周期 | 核心交付 |
|-------|------|------|---------|
| Phase 1 | MVP 自助查询 | 2 周 | 设备/库存/价格查询 + 单轮对话 |
| Phase 2 | 订单与同步 | 2–3 周 | 订单全流程 + 飞书双向同步 + 多轮对话 |
| Phase 3 | 智能与管控 | 2–3 周 | RAG 知识库 + RBAC + 监控告警 |
| Phase 4 | 运营与优化 | 2–3 周 | 推荐 / 评价 / 营销 / 性能优化 |

### 4.2 Phase 1 — MVP 自助查询 ✅

**目标:** 让客户能自助查到"有什么设备、还有没有货、租多少钱",验证 AI 对话查询的基本闭环。

**功能清单:**

1. 设备查询 — 列表、详情、规格、配置
2. 库存查询 — 按日期区间查可用数量(**需正确实现按日期占用**)
3. 价格计算 — 阶梯折扣 + 季节折扣
4. 单轮对话 — 意图识别后直接返回查询结果,暂不追踪上下文
5. 基础权限 — 区分 C 端 / B 端的数据可见范围

**不包含:** 下单、支付、飞书写入、多轮对话、RAG。

**价值:** 最小成本验证"AI + 数据查询"的体验,为后续下单流程打地基。

### 4.3 Phase 2 — 订单与同步 ✅

**目标:** 打通从下单到归还的订单主流程,实现与飞书的双向数据同步。

**功能清单:**

1. 订单管理 — 创建、修改、取消、查询
2. 库存预留 — 30 分钟锁定与自动释放
3. 支付状态记录 — 人工确认收款后推进状态(**不接支付通道**)
4. 飞书双向同步 — AI 写入飞书 / 飞书变更回流,含冲突解决
5. 多轮对话 — 会话管理、上下文追踪、意图漂移检测
6. 人工接入 — AI 置信度不足或客户要求时转人工工单

**价值:** 这是产品真正产生业务价值的 Phase —— 订单自动化 + 数据不再两头维护。

### 4.4 Phase 3 — 智能与管控 ⚠️🔵

**目标:** 提升 AI 回答的准确性,完善权限与系统可观测性。

**功能清单:**

1. RAG 知识库 — 设备资料 / 租赁规则 / FAQ 的语义检索问答 ⚠️
2. 细粒度 RBAC — 5 角色权限体系(若 Phase 1 未做)⚠️
3. 监控告警 — 关键指标监控、分级告警、服务降级 🔵
4. 数据分析报表 — 业务 / 财务 / 用户基础报表 🔵

> ⚠️ **待确认**:RAG 需要业务方提供结构化的设备资料和 FAQ 语料。**没有语料,RAG 无从谈起。** 启动 Phase 3 前需先评估语料的完备度。

### 4.5 Phase 4 — 运营与优化 🔵

**目标:** 在核心业务跑通后,叠加增长和体验优化能力。

**功能清单(均为规划项,细节待定):**

1. 推荐系统 — 热门 / 个性化 / 相似设备推荐 🔵
2. 用户评价 — 评分、评论、内容审核 🔵
3. 营销管理 — 优惠券、促销活动、会员等级 🔵
4. 性能优化 — 缓存、异步、多语言 🔵

> 🔵 **规划说明**:Phase 4 的需求大多是基于通用电商/租赁平台的推测,**尚未与业务方逐项确认**。列在此处是为了展示完整蓝图,但不应视为已承诺的范围。建议在 Phase 3 结束后,根据实际运营数据重新评估哪些值得做。

## 5. 关键业务流程

### 5.1 完整租赁主流程

```
查询设备 → 确认价格/库存 → 创建订单(待支付)
   → 人工确认收款(已支付) → B端审核(已确认)
   → 仓库发货(已发货) → 客户签收(使用中)
   → 客户归还 → 仓库验收+损坏评估(已归还)
   → 费用对账 → 完成
```

> 🔵 **v2.1 默认流程（客户可见标签视角）** —— 在上面完整模型上做了简化与合并：
>
> ```
> 客人下单 → 商家审核中
>    →(商家审核通过: 确认收款+放行档期一步完成)→ 已确认档期(待发货)
>    →(商家上传快递公司+物流单号)→ 已发货
>    →(商家验收)→ 订单已完结
> ```
>
> 与完整模型的差异：①「确认收款」与「B端审核」在商家管理页合并为一次「审核通过」；②发货后不经「客户签收/客户归还」，直接由商家验收完结（新增 `shipped → completed` 转换）。

每个状态转换的精确定义、触发条件、操作角色见 Spec 的订单状态机章节。

### 5.2 三个关键异常流程

**库存不足:** 客户所选日期区间内任一天库存不够 → AI 主动提示并给出备选(改数量 / 改期 / 换型号)。

**超期未还:** 系统按日检测 → 自动计费滞纳金 + 分级催收 → 第 7 天转人工。

**设备损坏:** 仓库验收发现损坏 → 拍照 → 对照赔偿标准自动算扣款 → 通知客户 → 客户可申请复议。

## 7. 依赖与风险

### 7.1 外部依赖

| 依赖 | 用途 | 风险 |
|------|------|------|
| 飞书开放平台 API | 数据同步 | 接口限流、稳定性 |
| 快递物流 API | 物流追踪 | 多家快递接口差异 |

| LLM 服务 | 意图识别 / RAG | 成本、响应延迟、准确率 |

### 7.2 关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 飞书 API 不稳定导致数据不一致 | 高 | 同步重试 + 冲突解决 + 降级为只读 |
| 按日期库存逻辑实现错误 | 高 | Spec 明确定义 + 重点测试覆盖 |
| RAG 语料不足 | 中 | Phase 3 前先评估语料,不足则推迟 |
| AI 误判意图执行错误操作 | 高 | 置信度阈值 + 涉及金钱操作需二次确认 |
| 5 角色权限过度设计 | 中 | Phase 1 先简化,按需扩展 |
