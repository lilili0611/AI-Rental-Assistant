文档中的需求按确认状态分三类,请评审时重点关注后两类:

## 1. 产品背景与目标

### 1.1 业务现状与痛点

相机租赁业务当前依赖人工处理咨询、下单、库存登记、对账等环节。随着订单量增长,人工模式暴露出几个问题:客户咨询响应慢(尤其非工作时间)、库存信息分散在飞书表格中容易出错、订单状态靠人工跟进、损坏赔偿计算主观且易产生纠纷。

### 1.2 产品定位

一个**对接飞书数据、面向客户与内部员工的 AI 租赁管理助手**。客户侧提供 7×24 自助查询与下单;内部侧提供自动化的订单、库存、对账流程。核心是把标准化、重复性的业务流程交给 AI,把判断性、异常性的环节留给人工。

### 1.3 核心目标与成功指标

| 目标 | 衡量指标 | 目标值 (上线 3 个月后) |
|------|---------|----------------------|
| 提升响应效率 | 客户咨询平均响应时间 | < 10 秒 |
| 降低人工负担 | 标准流程 AI 自动处理占比 | > 80% |
| 减少库存差错 | 库存数据与实物不一致率 | < 2% |
| 保证数据一致 | AI 系统与飞书同步延迟 | < 1 分钟 |
| 控制赔偿纠纷 | 因赔偿标准引发的客诉率 | 环比下降 50% |

> ⚠️ **待确认**:上述目标值为基于行业经验的建议值,需结合当前实际业务量与人力成本校准。

## 2. 用户与角色

### 2.1 用户分类

系统服务两大类用户,权限与功能边界严格区分:

**C 端 — 租赁客户**
自助查询设备、库存、价格;创建和管理自己的订单;追踪物流。仅能访问与自己相关的数据,无法触碰内部库存配置、定价规则、其他客户信息。

**B 端 — 内部员工**
按职能划分为 5 种角色(详见下表),通过飞书或后台管理订单、库存、客诉、财务。

### 2.2 B 端角色权限矩阵

| 角色 | 核心职责 | 关键权限 |
|------|---------|---------|
| 销售员 | 接单、报价 | 查看全部数据、创建订单、生成报价 |
| 仓库管理 | 出入库 | 确认库存、发货操作、验收归还、损坏拍照记录 |
| 财务审核 | 资金 | 审核订单、确认收款、处理退款、生成对账单 |
| 客服 | 售后 | 处理投诉、生成工单、授权补偿 |
| 系统管理员 | 全局 | 全部操作、用户与权限管理、系统配置 |

> ⚠️ **待确认**:5 角色划分是基于通用租赁业务的设计。如果当前是小团队(例如老板 + 1-2 名员工身兼多职),Phase 1 可简化为"客户 / 员工 / 管理员"三级,5 角色推迟到团队扩大后。**这一点直接影响 Phase 1 的权限模块工作量,请优先拍板。**

### 2.3 认证方式

✅ **已确认**:用户认证优先使用**手机号**(短信验证码)。实名认证(身份证)作为租赁前置要求,涉及押金和设备安全。

## 3. 业务规则 (核心,已确认)

本章是产品的业务内核,大部分已由业务方确认,是 AI 行为和系统计算的依据。

### 3.1 定价规则 ✅

价格由"基础租金 × 天数折扣 × 季节折扣 − 优惠券"构成。

**天数阶梯折扣:**

| 租期 | 折扣 |
|------|------|
| 1–3 天 | 原价 |
| 4–7 天 | 85 折 |
| 8–30 天 | 70 折 |
| 31 天以上 | 60 折 |

**季节折扣:** 按月配置(例:9 月份额外 90 折)。季节折扣与天数折扣**叠加相乘**。

> ⚠️ **待确认**:季节折扣的具体月份和力度需要业务方提供完整的年度日历,目前只有 9 月一个示例。

### 3.2 库存规则 ✅(但实现需修正)

✅ **已确认**:同一设备的不同配置(如「R5 + 16-35mm」与「R5 + 70-200mm」)**分别独立计库存**。

**库存占用是按租赁日期区间计算的** —— 这一点至关重要:某台设备在 9 月 1–3 日被租出,不影响它在 9 月 5 日的可用性。系统判断"某日期区间是否有货",必须检查该区间内**每一天**的占用情况。

> 🔴 **复盘提示**:Phase 1 已有代码采用的是全局计数(总数 − 已租 − 已预留),**无法正确处理按日期的库存**。这是 Spec 必须重新定义、开发必须修正的关键点。

**库存预留机制:** 客户下单后,系统预留库存 30 分钟。30 分钟内未完成支付确认,预留自动释放,库存回归可用。

### 3.3 支付规则 ✅(纠正前期设计)

✅ **已确认**:**支付由人工/第三方处理,AI 不自动扣款。**

这意味着系统中的"支付"环节是这样的:

1. AI 创建订单,状态置为"待支付"
2. 客户通过线下转账 / 第三方收款码 / 人工引导等方式付款(**不在本系统内完成**)
3. 财务或销售在飞书 / 后台**手动确认收款**,订单状态推进到"已支付"

> 🔴 **复盘提示**:前期设计文档与代码中出现的"生成支付二维码""支付宝/微信异步回调"等内容**与此确认矛盾,应全部移除**。系统只做支付状态的记录,不做支付通道的对接。

### 3.4 损坏赔偿规则 ✅(区间需校准)

赔偿按损坏类型和程度,以"扣除押金的百分比"计算。完整标准见下:

**机身/镜身划痕掉漆:**

| 程度 | 扣除 |
|------|------|
| ≤ 2mm | 押金 2% |
| 2–10mm | 押金 3% |
| 10–30mm | 押金 10% |
| > 30mm | 押金 15% |

**机身/镜身磕碰磨损:**

| 程度 | 扣除 |
|------|------|
| ≤ 1mm | 押金 3% |
| 1–3mm | 押金 5% |
| 3–5mm | 押金 10% |
| 5–10mm(外壳裂痕 ≤ 5mm) | 押金 15% |
| 10–20mm(外壳裂痕 ≤ 10mm) | 押金 20% |
| ≥ 20mm 凹痕 / ≥ 10mm 裂痕 | 押金 100% |

**镜片磨损:**

| 程度 | 扣除 |
|------|------|
| ≤ 1mm | 押金 5% |
| 1–5mm | 押金 10% |
| 5–10mm | 押金 20% |
| > 10mm 或出现镜片凹坑 | 押金 100% |

**其他:**

| 情况 | 扣除 |
|------|------|
| UV 镜变形或严重磨损 | 同 UV 镜价格 |
| 镜头卡口变形、对焦等性能故障 | 官方维修价 |
| 相机首次拆修 | 额外 10% 押金 |

> ⚠️ **待确认**:原始表格中部分区间边界存在歧义(例如"≤2mm"与"2–10mm"在 2mm 处、磕碰磨损里"≤3mm 全押金5%"与"3-5mm"的衔接)。Spec 中已将其整理为**左开右闭、无重叠**的精确区间,但需要业务方对照实物赔偿习惯做最终确认。

### 3.5 超期规则 ✅

租期结束未归还,按"超期天数 × 日租金 × 10%"计算滞纳金。处理节奏:第 1–3 天提醒并开始计费,第 7 天起进入催收流程。

### 3.6 取消规则 ✅

| 场景 | 费用 |
|------|------|
| 未支付,下单后 48 小时内 | 免费取消,释放库存 |
| 已支付,发货前 | 手续费 = 订单金额 10%,退还余额 |
| 已发货 | 不支持直接取消,需走拒收 / 人工 |

### 3.7 服务时效承诺 (SLA) ⚠️

| 环节 | 承诺 |
|------|------|
| 订单审核 | 24 小时内 |
| 发货 | 确认后 1–2 个工作日 |
| 验收 | 归还后 3–5 个工作日 |
| 对账 | 验收后 5 个工作日内 |
| 退款 | 申请后 3–5 个工作日 |

> ⚠️ **待确认**:SLA 是对客户的正式承诺,违约需补偿。请业务方确认这些时效是否符合实际履约能力,避免承诺过高。

## 4. 功能范围与 Phase 规划

本章是 PRD 的核心,按 Phase 拆解完整产品蓝图。每个 Phase 标注其功能、价值和确认状态。

### 4.1 总览

| Phase | 主题 | 周期 | 核心交付 |
|-------|------|------|---------|
| Phase 1 | MVP 自助查询 | 2 周 | 设备/库存/价格查询 + 单轮对话 |
| Phase 2 | 订单与同步 | 2–3 周 | 订单全流程 + 飞书双向同步 + 多轮对话 |
| Phase 3 | 智能与管控 | 2–3 周 | RAG 知识库 + RBAC + 监控告警 |
| Phase 4 | 运营与优化 | 2–3 周 | 推荐 / 评价 / 营销 / 性能优化 |

### 4.2 Phase 1 — MVP 自助查询 ✅

**目标:** 让客户能自助查到"有什么设备、还有没有货、租多少钱",验证 AI 对话查询的基本闭环。

**功能清单:**

1. 设备查询 — 列表、详情、规格、配置
2. 库存查询 — 按日期区间查可用数量(**需正确实现按日期占用**)
3. 价格计算 — 阶梯折扣 + 季节折扣
4. 单轮对话 — 意图识别后直接返回查询结果,暂不追踪上下文
5. 基础权限 — 区分 C 端 / B 端的数据可见范围

**不包含:** 下单、支付、飞书写入、多轮对话、RAG。

**价值:** 最小成本验证"AI + 数据查询"的体验,为后续下单流程打地基。

### 4.3 Phase 2 — 订单与同步 ✅

**目标:** 打通从下单到归还的订单主流程,实现与飞书的双向数据同步。

**功能清单:**

1. 订单管理 — 创建、修改、取消、查询
2. 库存预留 — 30 分钟锁定与自动释放
3. 支付状态记录 — 人工确认收款后推进状态(**不接支付通道**)
4. 飞书双向同步 — AI 写入飞书 / 飞书变更回流,含冲突解决
5. 多轮对话 — 会话管理、上下文追踪、意图漂移检测
6. 人工接入 — AI 置信度不足或客户要求时转人工工单

**价值:** 这是产品真正产生业务价值的 Phase —— 订单自动化 + 数据不再两头维护。

### 4.4 Phase 3 — 智能与管控 ⚠️🔵

**目标:** 提升 AI 回答的准确性,完善权限与系统可观测性。

**功能清单:**

1. RAG 知识库 — 设备资料 / 租赁规则 / FAQ 的语义检索问答 ⚠️
2. 细粒度 RBAC — 5 角色权限体系(若 Phase 1 未做)⚠️
3. 监控告警 — 关键指标监控、分级告警、服务降级 🔵
4. 数据分析报表 — 业务 / 财务 / 用户基础报表 🔵

> ⚠️ **待确认**:RAG 需要业务方提供结构化的设备资料和 FAQ 语料。**没有语料,RAG 无从谈起。** 启动 Phase 3 前需先评估语料的完备度。

### 4.5 Phase 4 — 运营与优化 🔵

**目标:** 在核心业务跑通后,叠加增长和体验优化能力。

**功能清单(均为规划项,细节待定):**

1. 推荐系统 — 热门 / 个性化 / 相似设备推荐 🔵
2. 用户评价 — 评分、评论、内容审核 🔵
3. 营销管理 — 优惠券、促销活动、会员等级 🔵
4. 性能优化 — 缓存、异步、多语言 🔵

> 🔵 **规划说明**:Phase 4 的需求大多是基于通用电商/租赁平台的推测,**尚未与业务方逐项确认**。列在此处是为了展示完整蓝图,但不应视为已承诺的范围。建议在 Phase 3 结束后,根据实际运营数据重新评估哪些值得做。

## 5. 关键业务流程

### 5.1 完整租赁主流程

```
查询设备 → 确认价格/库存 → 创建订单(待支付)
   → 人工确认收款(已支付) → B端审核(已确认)
   → 仓库发货(已发货) → 客户签收(使用中)
   → 客户归还 → 仓库验收+损坏评估(已归还)
   → 费用对账 → 完成
```

每个状态转换的精确定义、触发条件、操作角色见 Spec 的订单状态机章节。

### 5.2 三个关键异常流程

**库存不足:** 客户所选日期区间内任一天库存不够 → AI 主动提示并给出备选(改数量 / 改期 / 换型号)。

**超期未还:** 系统按日检测 → 自动计费滞纳金 + 分级催收 → 第 7 天转人工。

**设备损坏:** 仓库验收发现损坏 → 拍照 → 对照赔偿标准自动算扣款 → 通知客户 → 客户可申请复议。

## 6. 非功能性需求

| 维度 | 要求 |
|------|------|
| 性能 | API 响应 P99 < 3 秒 |
| 并发 | 支持 100+ 并发(初期),可水平扩展 |
| 可用性 | 核心查询服务可用性 > 99.5% |
| 数据安全 | 身份证、地址等敏感信息加密存储(AES-256) |
| 隐私合规 | 符合个人信息保护相关法规,支持数据导出与删除 |
| 数据一致性 | 飞书同步采用时间戳冲突解决 + 乐观锁 |
| 可观测性 | 关键操作审计日志,不可篡改 |

## 7. 依赖与风险

### 7.1 外部依赖

| 依赖 | 用途 | 风险 |
|------|------|------|
| 飞书开放平台 API | 数据同步 | 接口限流、稳定性 |
| 快递物流 API | 物流追踪 | 多家快递接口差异 |
| 短信服务 | 手机号认证 | 到达率、成本 |
| LLM 服务 | 意图识别 / RAG | 成本、响应延迟、准确率 |

### 7.2 关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 飞书 API 不稳定导致数据不一致 | 高 | 同步重试 + 冲突解决 + 降级为只读 |
| 按日期库存逻辑实现错误 | 高 | Spec 明确定义 + 重点测试覆盖 |
| RAG 语料不足 | 中 | Phase 3 前先评估语料,不足则推迟 |
| AI 误判意图执行错误操作 | 高 | 置信度阈值 + 涉及金钱操作需二次确认 |
| 5 角色权限过度设计 | 中 | Phase 1 先简化,按需扩展 |

## 8. 待决策清单 (评审重点)

以下问题需要在评审会上拍板,直接影响开发范围和优先级:

1. **权限模型**:Phase 1 用三级简化权限还是直接上 5 角色?
2. **季节折扣**:能否提供完整的年度折扣日历?
3. **SLA 时效**:第 3.7 节的承诺是否符合实际履约能力?
4. **赔偿区间**:第 3.4 节整理后的精确区间是否符合实际赔偿习惯?
5. **Phase 3/4 范围**:RAG 语料是否具备?Phase 4 哪些功能真正要做?
6. **支付方式**:人工收款具体走什么渠道(线下转账 / 收款码 / 其他)?需要系统记录哪些支付凭证?
